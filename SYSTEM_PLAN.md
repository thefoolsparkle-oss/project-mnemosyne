# 忆界树 / Project Mnemosyne 接力计划

这份文档给项目作者和下一轮开发对话使用。换一个聊天窗口后，先读本文件，再读 `README.md`，即可接上当前方向。

## 项目定位

忆界树是一个长期多人格 AI 陪伴平台。当前阶段重点不是固定几个预设人格，而是让用户注册后，通过“你想和什么样的人交谈？”创建初始对象。这个对象后续会被聊天、记忆、用户画像、关系状态和经过审查的人格修订逐步塑造。

核心方向：

- 用户不是填设定表，而是在聊天中逐渐得到更合适的陪伴对象。
- 忆界树不是单纯聊天工具，而是长期关系养成产品。用户需要持续感到“这个对象因为我而变化、记得我、回应我、靠近我”。
- 只让用户舒适还不够，系统必须逐步提供吸引力和反馈感，否则会退化成无聊的问答窗口。
- 用户当然知道对象由 AI 驱动，但人格自己不能开口自称 AI、模型、机器人或程序。
- 人格要像一个有自主性、能自然聊天的人，而不是解释自己在执行系统 prompt。
- 记忆系统已经第一阶段收束，后续不要继续无限加记忆功能；第二阶段人格成长与反馈体验已完成功能闭环并转入观测维护，当前主线开始进入第三阶段轻表达系统。
- 多 agent 数量不预设，随着系统成熟逐步出现。不要为了做架构而硬拆 agent。
- LLM 不固定 DeepSeek，未来要支持多供应商、多模型按任务难度、成本和重要性路由。
- 远期会扩展到人物形象、头像/立绘/表情、声音、语音聊天，甚至视频电话或实时形象驱动，但这些属于后阶段能力，需在人格成长与反馈链路稳定后展开。
- 多平台功能要平行推进：网页桌面端、手机端/PWA、后续桌面壳和原生移动端不能各做一套割裂产品。核心交互、主界面、未读状态、资料页、聊天体验和账号数据必须共享同一套后端能力与产品结构，再按设备做布局适配。
- 未来要建立主动联系系统：人格可以在合适的时候主动给用户发信息，增强关系生命感和用户兴趣；但必须先取得用户许可，并根据长期反馈学习频率和类型，不能变成骚扰或营销推送。

## 已完成重点

基础：

- FastAPI 后端。
- SQLite 本地数据库 `data/app.db`。
- 注册、登录、退出、session cookie。
- 第一个注册用户自动成为 admin。
- 普通端和管理端隔离。
- 登录保持 30 天。
- 同浏览器多账号测试模式：普通登录仍使用共享 Cookie；登录页可选“此标签页独立登录”，已登录页面也可点“切换账号（仅本页）”，以标签页独立凭证同时测试不同账号；标签页凭证只覆盖该页请求，不替换其他标签页的普通登录。
- 游客模式第一版：登录页可直接进入“游客体验（3天）”。游客账号标记为 `is_guest`，有 `guest_expires_at`；三天后访问或服务启动时自动删除游客用户及其关联数据，并清理 `data/uploads/<user_id>`。游客不占用“第一个正式用户自动成为 admin”的资格。
- 游客可转正式账号：游客资料弹窗里可以设置账号、密码和昵称，转正后保留当前人格、会话、消息、记忆和头像，`is_guest` 清零，session 切回 30 天。

人格：

- 删除旧固定人格残留。
- 人格创建改成 Forge 向导：“你想和什么样的人交谈？”
- 支持少量提示 chip + 自由描述 + 直接生成。
- `persona_forge.py` 生成初始人格轮廓；`persona_namer.py` 的独立 Name Crafter 根据用户原话、人格轮廓与该用户已有名字生成初始名称。
- 初始名字不再来自固定随机池：用户明确指定名字时直接尊重指定；未指定时由 Name Crafter 命名并避开已有对象重名；命名模型未取得合格名字时不保存“未命名”对象，而是在原创建页保留输入并允许再次生成或由用户亲自命名。
- 数据库启动不再把历史占位名称强行改成某个固定人格名；已经存在的人格名称只由用户编辑或后续明确的人格演化流程修改。
- 人格字段已扩展：心理适配、成长备注、外貌描述、期望形象、头像 URL。
- 人格版本可追踪。

聊天体验：

- 主界面和聊天侧栏均以人格为唯一入口，不再把同一人格的不同聊天段落铺成重复对象；聊天段落在对象内的“记录”面板管理。
- 聊天页像 IM：左右头像、气泡、会话列表、顶部对象信息、底部输入框。
- 聊天记录会按日期显示轻量分隔条，例如今天、昨天或具体日期。
- 消息悬停时提供轻量复制按钮，复制原始整条消息文本。
- 空会话提供少量开场短句按钮，点击后只填入输入框，不自动发送。
- 人格资料不常驻，点击头像或“资料”才弹出。
- 人格资料编辑已从空壳改为有效改动：名字、摘要、关系定位、说话方式、外貌参考、期望形象会写回数据库，重建人格 prompt，并写入人格版本历史。
- 用户和人格头像支持本地图片上传，图片保存到 `data/uploads` 并通过 `/uploads/...` 访问；URL 只作为可选图片链接方式。
- 聊天运行时会额外注入当前人格资料作为强约束，确保用户编辑过的名字、摘要、关系定位、说话方式等立即反馈给聊天 AI。
- 聊天上下文末尾已增加 Final persona lock，用当前人格名、关系定位、说话方式压住旧历史消息、旧摘要或旧记忆里的自称。
- 如果人格改名，聊天上下文会根据人格版本历史把旧 assistant 自称和旧上下文里的旧名改写为当前名，避免历史消息把模型重新带回旧名字；数据库原始聊天记录不被篡改。
- 新会话标题以人格名初始化，不再用第一句用户消息当标题；用户重命名的聊天段落会在对象的“记录”面板中真实显示。
- Forge 和聊天规则已去掉默认恋人口径：界面只给少量宽泛方向；具体关系只有用户选中或在自由描述里正向明确写出时才落库，模型不能自行补成恋人/陪伴者，“不要像恋人”这类否定表达也不会反向污染关系。未指定关系时数据保持“关系未定”，界面以“刚刚认识”自然呈现。
- 已做系统级身份污染清理：Forge 不再保存模型生成的原始 prompt，而是统一用本地干净 prompt；聊天运行时不再反复注入“AI/模型/机器人”身份词；历史 persona、persona_versions、assistant messages、context traces 和 persona revision suggestions 已清理。
- 未来防线已覆盖主要入口：用户编辑人格资料、人格创建、人格调整建议、记忆抽取、用户画像、会话摘要、聊天上下文注入和最终回复输出都会经过身份污染清理。
- 用户资料不常驻，点击用户头像/昵称弹出。
- 页面整体固定高度，左侧列表和聊天内容各自内部滚动。
- 发送后不再显示“正在输入...”气泡；AI 回复回来后直接出现。
- 回复生成失败会显示为非人格的状态提示，不让人格开口解释“服务不稳定”；失败状态保存在原用户消息上，刷新或重新进入聊天后仍可原地重试；每次发言携带客户端消息 ID，临时断网或隧道抖动后的再次尝试会复用同一句，不重复写入聊天或记忆；失败提示已能安全区分限流、超时、模型服务不可用和环境变量/API Key 配置问题，不向普通端暴露真实密钥或堆栈。
- 普通端发送等待状态已分阶段显示：短等待时说明消息已送出，较慢时提示模型仍在响应，长等待时告知消息不会丢且失败会原地给出重试，降低“黑箱卡住”的体感。
- 会话滚动摘要刷新不再阻塞已生成的聊天回复：`/api/chat` 先返回人格答复，再由后台任务刷新 `conversation_summaries`；摘要模型超时不会把已经完成的这一轮回复一起拖到超时。
- 用户消息一旦成功落库即刷新会话最近活动时间；即使本轮回复失败，主界面排序不会停留在旧对话，对象卡片与聊天记录预览也会标明“回复未送达”。
- 会话可重命名、归档和恢复。归档不是真删除，普通列表默认隐藏归档会话。
- 删除人格第一版已实现：资料编辑页底部危险操作区要求输入人格当前名字二次确认；删除采用软删除，把人格状态改为 `deleted`，普通端隐藏人格与相关会话，聊天接口也不再允许读取/续聊该对象；主界面/创建页可从“已删除”面板恢复对象与保留的聊天记录；人格 JSON 数据导出第一版已完成；已删除面板支持彻底清除，需再次输入名字确认，清除后人格资料与单聊数据不可恢复，群聊旧消息会脱敏保留时间线。
- 会话列表支持前端本地搜索，可按会话标题、最后消息、人格名等过滤；搜索时保留焦点，并提供轻量清空按钮。
- 会话支持置顶/取消置顶，置顶状态写入数据库并在列表中优先显示。
- 输入框会按内容自动增高，并按当前人格/会话保留本地草稿；发送成功后清除草稿。
- 刷新或重新登录后打开最近会话，并自动滚动到底部。
- 前端会记录最后打开的人格和会话，刷新/重新登录优先恢复到上次聊天现场。
- 当前前端会把较长的 assistant 回复按自然标点拆成多条短气泡展示；数据库仍保留原始单条回复。后续再升级为真正的分段节奏发送或流式。
- 普通端第一版对象状态层已加入：聊天顶部和未开聊人格项会显示关系、说话方式和版本，让对象存在感比单纯名字/摘要更强。

- 完成机制、过期机制、偏好冲突、称呼变化。

- Conversation rolling summary。
- SQLite JSON embedding 版 Memory RAG。
- 上下文追踪（内部数据 `Context Traces`）记录每次实际注入给模型的上下文，管理台可展开查看记忆、探索防重复/维度覆盖策略和资料按需使用/日期环境。
- Memory Cost Mode：eco / balanced / deep。

管理台：

- 用户列表、人格列表。
- 记忆状态变量、动态状态、用户画像、会话摘要。
- 记忆审校、记忆冲突、人格调整建议、上下文追踪。
- 评测按钮：Core / Chat / Resolve / Expiry / Policy / Live。
- 人格成长面板：查看哪些记忆推动人格变化、人格调整建议、版本历史。
- 第二阶段核心链路第一版已完成：普通端资料页新增克制的“相处痕迹”，主界面仅对尚未查看的已审核变化显示一次轻提示；人格版本新增变化类型、来源建议编号与变化摘要；管理台可比较候选建议和历史版本差异，并通过基线版本锁防止过期建议覆盖新改动。
- 用户在聊天中明确提出表达或相处方式调整时，规则提取会仅依据本轮触发线索形成一条保守记录；若只调整回应方式/成长备注，受限自动审核代理会即时应用，本轮回复即重读更新后的说话方式，并记录执行方与理由；若用户明确设置名字或关系，例如“把你的名字改成小舟”“以后你就是我的女朋友吧”“我以后叫你阿澄吧”，也会立即更新并保留版本记录；若用户明确结束当前关系，例如“我们分手吧”“以后你别再当我的女朋友了”或“我们先不要定义关系了”，则将关系直接回到中性的“关系未定”并留档，不替用户推断新的定位。含糊、推断或带否定前缀的设置句（例如“不要把我们的关系改成朋友”）不会改变核心资料，重复已满足要求不会制造新人格版本。
- 人格调整审核决定已有审计记录：管理台在应用或忽略建议时可填写备注，候选持久化处理管理员、处理时间与处理原因，未应用的建议也能解释最终判断。
- 管理台刷新人格数据后会保留仍存在的当前选中对象，应用建议后可以原地查看审核记录与版本变化，不会跳回第一个人格。

表达层：

- AI 文本中的括号动作不应直接显示。
- 已有第一版 expression layer：
  - 从 `（托腮）`、`【轻笑】`、句中动作括号中抽取 expression label。
  - 清洗后的文本进入聊天气泡。
  - expression label 存入 `message_expressions`。
  - 前端先用小标签显示。
- 第三阶段第一步已落地受限结构化轻表达：聊天人格仅在确实需要时可在回复末尾附带一个白名单程序标签，例如 `[[expression:mood:微笑]]` 或 `[[expression:tone:轻声]]`；后端剥离并校验后沿用 `message_expressions` 展示，无效标签不进入界面。
- 表达标签目前只允许少数 `mood` / `tone` / `gesture` 值，不接贴纸、不触发额外模型，也不允许用标签替代正常文字回应；标签从轻量表达资源目录派生，目录包含展示文案、图标、分组、风险级别、强度、排序、兼容别名和模型用途说明，并会生成带资源元信息的聊天 prompt 白名单；当前资源类型先是 `text_badge`，后续再替换为贴纸、头像表情或动作资源；括号动作提取继续作为旧数据与违规输出兼容路径，但只会按启用资源别名归一到白名单标签。
- 轻表达频率控制第一版已加入：聊天 prompt 会看到当前会话近期表达节奏；若上一条回复刚展示过表达，本轮标签被抑制，同名标签会按资源目录的冷却轮数避免重复落库或显示；资源目录标为中风险的表达在近期已有任何轻表达时会额外抑制。
- 用户侧轻表达偏好已加入三档：用户在聊天中明确说“以后别发表情了 / 不要动作标签”会写入 `expression_preferences.mode = off`，当轮起不再输出或展示表达标签；说“表情少一点 / 轻表达克制一点”会写入 `subtle`，近期展示过轻表达时继续抑制，只有明显需要承接时才允许；明确说“现在可以发表情了”会恢复为 `normal`。联系人资料页也提供关闭/克制/正常三档，不必靠聊天指令完成设置。
- 启动时会对旧 assistant 消息做一次轻量整理，清除历史括号动作和明显身份泄漏，并补写 expression label。
- 第三阶段验证脚本：`py scripts/verify_phase3_expression.py` 检查表达资源目录、协议提示、白名单解析、无效标签清理、括号兼容提取、消息持久化回读、连续展示冷却、近期同标签抑制和轻表达三档偏好。
- 后续再接真实表情包、头像表情、Live2D 或动作资源。

## 近期用户指出的问题

这些是后续开发必须尊重的判断：

- UI 目前仍然粗糙，配色单调可以后面改，但布局和交互要先稳。
- 不要让资料页常驻，资料应像 QQ/微信一样点击头像后出现。
- 聊天框、侧栏、输入框要固定，内容多了只滚动聊天内容。
- 刷新或重新登录不能停在聊天顶部，必须默认定位到底部。
- 不要“正在输入...”气泡。
- 不要让 AI 回答里出现 `(动作)`、`（神态）`、`【表情】` 这类文本。
- 人格绝对不能自称 AI、模型、机器人、虚拟人格、专属 AI 恋人或聊天程序。
- 用户知道它是 AI，但沉浸感要求人格自己不要这样说。
- 表情/动作管理需要小 agent，但属于第三阶段表达系统，不在当前第二阶段展开。
- 现在不要依赖本地模型，Ollama 暂时不是重点。
- 不要预设固定小 agent 数量，随着系统完善自然增加。
- 不要只依赖 DeepSeek，长期要支持市面所有可用 LLM 的任务分配。

### 普通端 UI 设计方向

短期不要做营销 landing page，第一屏仍然是可用聊天体验。但视觉和结构要逐步从“工程界面”转向“关系产品”：

- 聊天是主舞台，左侧列表只是入口，不能抢占大量空白。
- 人格头像、名字、关系摘要、最近状态要形成明确对象存在感。

- 养成反馈先轻量出现，例如“最近更熟悉你了”“记住了一件事”“性格有一点变化”，但不要做夸张等级条。

- 视觉上可以参考成熟 IM 的稳定布局，也参考 AI companion 产品对头像、形象、记忆、关系连续性的强调。
- 第一阶段已修密度、层级、按钮位置和移动端可用性；完整主题、色彩和品牌视觉稍后统一重做。
- 移动端主界面是关系产品的入口，不只是响应式会话列表。人格卡片应承载“这个对象在等你/有新内容/最近状态变化”的反馈感，但第一版先做未读红点和最近消息。

### 第一阶段功能扩展：多人格群聊（文字群聊第一版）

群聊只是聊天体验的一种入口形态，归入第一阶段“文字聊天基础”的扩展维护项，不单独作为新阶段，也不改变当前第三阶段表达系统主线。

用户反馈一对一聊天容易无聊，因此增加“一个用户 + 多个人格”的群聊模式。这个方向不要简单做成多个单聊回复拼接，而要有群聊调度层。

第一版后端骨架和普通端入口已落地：新增独立群聊表、群成员表、群消息表和 API；`Group Router` 会先判断本轮 0-2 个发言人格，再分别生成人格短回复。普通端主界面已有群聊入口、建群选择人格、群聊卡片、群消息渲染、历史群聊恢复入口和群设置卡片。

核心原则：

- 群聊会话独立于现有单人格 `conversations`，避免破坏已经稳定的单聊、未读、历史记录和人格成长链路。
- 一个群聊包含多个人格成员，每个成员有入群顺序、是否活跃、群内昵称/角色和最近参与状态。
- 用户发言后先由调度器判断哪些人格应该接话：可以无人接、一人接、多人接，也可以某个人格接另一个人格的话。
- 人格可以在低频条件下主动抛话题，但必须受节奏限制，避免刷屏和抢用户注意力。
- 群聊消息需要记录真实 speaker：用户、某个人格、系统提示或未来的群聊调度事件。
- 群聊上下文要给每个人格看到“群内最近发生了什么”和“自己与其他人格的关系/立场”，而不是只看到用户和自己的单聊历史。

已落地范围：

- 新增 `group_conversations`、`group_members`、`group_messages`，暂不复用单聊 `conversations.persona_id`。
- 创建群时选择 2-6 个人格，默认标题可用成员名组合。
- 用户发言后由 `Group Router` 输出本轮发言计划，例如沉默、单人回复、双人连续回复、接某个人格的话。
- 每个被选中的人格再用自己的 prompt、人格资料和群聊上下文生成一条短回复。
- 普通端可创建群、打开群、查看群消息、归档/恢复群、修改群名、置顶/取消置顶并查看成员。

### 第二阶段：人格成长与反馈体验（功能完成，转入观测维护）

目标：让用户逐渐感受到对象确实因长期相处而变化，同时让变化有依据、可审查、可追踪，不能让人格无缘无故漂移。

- 用户侧轻量反馈：把“更熟悉你了”“记住了一件重要的事”“相处方式发生了轻微变化”等反馈自然放进资料细节或适当入口，不做夸张等级条和频繁系统弹窗。
- 管理员成长面板：能看到哪些记忆、状态变化或聊天反馈推动了人格调整建议。
- 人格调整证据流程（内部模块 `Sculptor`）：展示调整依据与影响范围；受限自动审核代理落实明确的低风险回应方式要求；明确的聊天改名/关系设置指令自动更新核心资料并留档，模糊或推断表达不更新，也不产生管理员日常审批负担。
- 人格版本证据链：版本可追踪、可比较，能够回答“改了什么、为什么改、依据是什么、何时应用”。
- 变化边界：不得因单句随口对话大幅改写核心人格，不得把推断误写为用户明确期待，不得破坏当前关系定位和身份污染防线。
- 跨端一致：桌面与手机都能承载必要的人格变化反馈；第一阶段移动端维护与本阶段功能同步推进。

当前已落地的第二阶段第一版闭环：

- 普通端视觉主题根据实机观感调整为白底浅色方向：登录、主界面、聊天和资料弹窗使用白色/浅灰卡片层次，交互强调改为灰蓝色，仅为变化提醒和危险操作保留必要的语义色。
- 普通端联系人资料中加入“相处痕迹”卡片：没有足够依据时保持克制提示；存在记忆、当前状态或已审核人格调整时显示概括反馈；最近审核变化仅基于普通端可见字段生成“回应方式更贴近你的偏好”等摘要并标出确认时间；同时轻量提示用户可以直接在聊天中表达相处偏好，不直接展示后台原始记忆或审核备注。
- 主界面人格入口与聊天顶部加入“有变化”轻提示及已读状态：只为已经审核应用的 `sculptor_review` 版本显示（包含受限自动审核代理通过的低风险变化）；首页与聊天顶部的提示均可直接打开资料中的“相处痕迹”，查看后消退；不把待审建议或普通记忆做成通知。
- 用户主动写下的相处偏好作为当前运行时指导，不因人格版本变化而过期；用户可在“相处痕迹”直接更新或停止该指导。
- 新产生的人格版本区分 `initial_forge`、`user_profile_update`、`sculptor_review`，审核应用的版本持久化来源建议编号和变化摘要；建议记录保存落地版本号。
- 每条新人格调整建议保存生成时的 `base_version`；审批前若人格被用户编辑或其他建议更新，旧建议会显示过期且后端拒绝应用。
- 管理端显示建议依据概览、候选字段差异、历史版本相邻差异、审核建议关联和落地版本，可从版本回到对应建议。
- 用户在聊天中明确说出“回复短一点”“不要说教”“少追问”等表达偏好时，系统会即时建立 `chat_feedback` 当前指导，并基于同一条经明确指令过滤的 `persona_feedback` 生成候选，不额外调用人格调整模型；否定要求或只是讨论某句偏好时，不产生指导或人格变化；明确说“回复详细一点这条不用了”等停止表达时，直接停用匹配内容及关联旧记忆，若该条还包含未取消要求，则用保留片段继续形成当前指导且留下替换证据。当字段差异只限 `speaking_style` 与 `growth_notes` 时，自动审核代理立即应用，并在同一次聊天生成中重载已更新人格；指导随后失效时，由 `guidance_reconcile` 审计版本确定性去掉已写入的陈旧回应依据。直接聊天中的明确反向要求也会替代旧活跃指导并归档其旧记忆，避免版本备注或召回内容继续压过用户新说法。用户明确说出“把你的名字改成小舟”“以后你就是我的女朋友吧”或“我以后叫你阿澄吧”时，另由确定性指令路径直接更新核心资料并写入 `explicit_core_update` 审计版本；明确说“我们分手吧”“以后你别再当我的女朋友了”或“我们先不要定义关系了”时，同一路径把当前关系重置为“关系未定”，不推断替代关系；“不要像恋人那样”与“不要把我们的关系改成朋友”这类只谈表达或否定改动的句子都不触发核心更新。已由当前状态满足的重复要求不会新增版本。
- 管理员应用或忽略候选时可记录审核备注；自动审核代理应用时也写入固定审计说明。`persona_revision_suggestions.decision_actor` 区分 `admin`、`review_agent` 与用户撤回操作，管理台会显示审核执行方，因此每次决定均保留可追溯理由。
- 普通端不会看到人格调整器的原始变化说明、触发记忆或管理员审核备注；资料页只根据审核前后公开人格字段显示克制的已确认变化摘要，避免把后台证据泄露成用户侧文案。
- 管理端通过用户级与人格级概览展示审计与遗留清理信息；选中人格后可让自动审核代理补处理旧版低风险聊天积压，也可批量忽略过期候选。新的日常偏好不进入此队列。
- 普通端可对最近一次已确认变化做一次轻量回传，选择“这样更合适”或“还想调整”；选择后者时可附一段具体说明。记录绑定具体 `sculptor_review` 版本，管理端可查看历史反馈与用户说明；具体调整文字会立即加入该版本对应的当前回应指导并在用户侧显示已处理，不再等待管理员逐条审批，也不会覆盖资料页主动偏好。
- 普通端可直接提交一段相处偏好，聊天中明确表达的回应要求也自动进入同一指导历史。请求作为本人可回看的 `persona_growth_requests` 保存并写入高优先级 `persona_feedback` 记忆，同时在聊天上下文中即时注入；`request_origin` 区分 `direct_entry`、`growth_feedback` 和 `chat_feedback`，`source_reviewed_version` / `source_message_id` 关联来源，`deactivation_actor` 与 `deactivation_reason` 保存用户停止、聊天停止或自动替代原因。兼容指导可同时生效；确定性识别出的反向要求会归档旧记忆并显示为被新指导替代，聊天中的明确停止会显示为已停止，组合条目则只将未取消内容续为新的当前指导。旧版本产生的 `profile_request` 待审候选会在数据库初始化时标记为由自动适配接管，不再形成管理员积压。
- 新识别的体验问题归入第二阶段持续打磨：用户画像只有极少线索时，多层记忆注入会放大某个偏好，使人格反复围绕同一件事展开而不是继续认识用户。实际聊天上下文靠近回复的位置现已增加探索策略，要求单条喜好仅在相关时使用、不得成为无关对话的固定话题；明确的兴趣偏好改口遵循最近一次表达，例如“喜欢原神 -> 现在不喜欢了 -> 现在又喜欢了”会同步更新 `Mirror`、旧版记忆与分层记忆，旧方向不再作为当前上下文注入；明确的话题边界与喜欢可同时存在，例如“喜欢原神，但别主动提原神”不会被喜欢信号冲掉，而“原神可以聊了 / 以后可以主动聊原神了”会自动归档旧边界并保留已解决审计，仍有效的“不喜欢原神”避开约束不会被误删；禁用称呼也支持用户直接改口撤销，“现在可以叫我宝宝了”只移除旧禁称而不误设默认称呼，“你可以叫我主人 / 以后叫我小禾”则作为明确首选称呼设置，同时释放同名禁称、归档旧首选称呼的 legacy 记忆并将替代证据标为已解决，避免新旧叫法同时进入聊天；分层记忆替代审计已改为写入提交后执行，避免明确改称呼时出现 SQLite 写锁。`user_insights.discovery_dimensions_json` 会累计用户已经明确透露的兴趣、日常节奏、价值/在意之事、安慰方式、边界、计划期待和相处期待覆盖，管理台画像可查看概览，回复策略在自然适合提问时优先尚不清楚的方向而不是反复挖掘已有点，但不得把覆盖缺口变成问卷；`user_insights.curiosity_feedback_json` 只在用户明确拒绝或邀请探索提问时记录态度，拒绝后暂停主动个人提问，直到用户明确重新邀请，管理台可看到当前谨慎/开放状态及累计次数；若最近几轮 assistant 回复已使用某个喜好、而用户本轮没有主动提及，则本轮明确压制再次硬带回该话题。因为用户知道这是 AI，提问可比现实陌生人更直接，但仍必须一次克制、允许跳过、拒绝后不追问，也不得由单条信息推断整体人格或关系需求。
- 用户资料并非未送入主聊天：对真实茵陈聊天追踪的检查确认，`nickname`、`gender`、`birthday` 等字段在用户问到之前就已经进入 profile prompt。失败点不是资料断链，而是主聊天缺少一种通用规则：用户要求查资料、询问已知事实或给出可由资料与日期解释的线索时，应使用相关字段直接回答，而不是继续猜测。第二阶段现已加入该按需核对规则及中性本地日期环境；查询资料时只答被问字段或解开当前线索所需的一个事实，不顺手追加未询问的记忆、兴趣或纪念日，也不把字段拿来反复表演熟悉感。管理台追踪可查看当轮规则；日期第一版使用运行服务器时区，后续面向异地用户时需增加用户时区字段或客户端时区同步。
- 为避免继续拿真实聊天碰运气验收，管理台新增独立资料使用评测：`profile_context_v1` 不消耗聊天模型额度，检查昵称/签名注入、按需核对触发和普通聊天的克制状态；`profile_live_answer_v1` 可选调用真实聊天模型，以昵称和签名为中性样本验证资料回答，并检查不会顺手扯入兴趣或避雷话题。生日只保留为曾暴露问题的真实例子，不作为评测中心或主动话题。
- 因早期缺乏真实成长数据，管理台新增专用演示沙盒：载入时创建可登录的独立样本账号以及完整的审核/反馈/主动偏好请求/候选链路，清除时级联删除样本；它只解决验收可见性，不替代真实交互验证。

第二阶段暂不扩张到：真正分批且可打断的短消息、完整表情资源、主动联系策略、多模型成本调度深化、生图、语音或视频。

后续按真实聊天逐步处理该问题：先观察探索规则、维度覆盖引导、显式提问态度和近期话题降重能否降低重复提及并保持自然，尤其关注拒绝措辞识别是否过窄或误触发；再考虑按账号保存时区；只有涉及离线或退出聊天后的主动发消息时，才进入第四阶段的主动联系系统。

### 第三阶段：表达系统（已启动：结构化轻标签）

当前先不做完整表情包系统：复用已有 `message_expressions` 与前端轻标签展示，为单聊和群聊人格回复增加至多一个经过白名单验证的结构化表达标签。模型正文仍保持自然对话，标签由程序剥离后单独呈现；表达显示已有连续冷却、资源级同标签冷却和中风险资源近期额外抑制；用户可在聊天中或联系人资料页把该人格的轻表达设为关闭、克制或正常；管理台已能查看当前资源目录、按资源分组折看启用数量、启用/禁用单个资源、编辑资源备注、调整资源冷却轮数、查看当前模式、最近单聊/群聊表达记录、窗口统计、标签计数和调参提示，且历史使用记录会附带资源当前启用状态、风险级别、分组、冷却和来源，用于后续调频率与审查资源；管理台审查建议第一版会把已禁用资源历史、中风险标签和高占比标签整理成结构化 `review_items`，前端可直接从建议提高冷却或补审查备注，也可一键把当前窗口中可应用的冷却建议批量写回资源设置；来源分布调参第一版已接入，统计窗口会拆分模型标签、选择器补充、括号兼容和未知来源，并在选择器占比过高时生成冷却/场景收紧建议；资源生命周期第一版已接入 `active/paused/archived` 字段，管理台可归档/恢复资源，归档资源会从普通端资源接口、模型白名单提示、结构化解析和括号兼容提取中同步移除，但管理端仍保留资源和历史记录用于审查；真实媒体资源承载层第一版已接入，资源目录稳定暴露 `media_url`、`thumbnail_url` 和 `alt_text`，普通端轻表达 pill 与管理台资源卡都能在未来资源 `asset_kind` 切到 image/gif/avatar_expression 时渲染缩略图；媒体 URL 配置第一版已接入，管理台可给单个表达资源写入 `asset_kind`、媒体 URL、缩略图 URL 和替代文本，也可清空后恢复为文本徽标；真实素材上传第一版已接入，管理员可直接给单个表达资源上传 jpg/png/webp/gif，后端统一校验 5MB 限制、写入 `/uploads/expression-assets/` 并自动绑定媒体 URL；媒体批量导入第一版已接入，管理台可粘贴数组或 `{items:[...]}` 批量写入表达资源媒体配置，后端逐条返回成功与失败，不因单条标签错误中断整批；媒体审核第一版已接入，批量导入素材默认待审，普通端不展示未批准媒体，管理台可通过或驳回并保留审查备注，批准后才进入普通端轻表达渲染；素材审查队列第一版已接入，管理台资源目录可按全部、待审、驳回、已通过筛选，并优先展示待审素材；素材来源记录第一版已接入，手动 URL、单张上传、批量导入都会写入来源类型和来源详情，管理台资源卡可追踪素材从哪里进入；表达选择小 agent 第一版已接入：当模型没有输出结构化标签时，程序会根据场景、人格风格、分组、冷却和用户偏好最多补一个白名单表达，不额外调用模型，也不会绕过禁用或近期抑制；群聊表达选择第一版已接入，群聊人格回复和自主续聊也会在模型无标签时复用同一套 selector、场景和冷却约束；单聊情绪场景频率策略第一版已接入：用户文本会被归入 `support_needed`、`playful` 或 `ordinary`，prompt 会注明当前可用资源分组，程序层也会阻止普通闲聊使用 support/care 类标签，并允许真正需要安慰时在冷却约束内使用 care 类表达；人格专属轻表达风格可审查设置第一版已接入：管理台可按人格显式设置 `restrained/playful/warm/neutral`、优先资源分组和避用标签，单聊和群聊运行时都会优先使用该设置，没有显式设置时仍从人格资料推断；风格设置历史审计第一版已接入，每次管理台修改都会记录事件，管理台会展示最近风格设置变化；风格批量建议第一版已接入，管理台会根据最近表达的主导分组生成可一键应用的人格风格建议；轻表达资源目录已经从展示列表推进为协议来源，包含分组、风险级别、强度、冷却、兼容别名和模型用途说明，聊天 prompt 白名单也会带上这些资源元信息；后续增加标签也应优先改资源目录；括号动作只作为清理和兼容路径，不鼓励模型继续输出。

场景分布观测第一版已接入，管理台会从选择器来源拆出普通、玩笑和安慰场景，便于继续调轻表达触发频率。
表达资源变更历史第一版已接入，启用/禁用、归档/恢复、冷却、媒体配置与媒体审核都会写入统一事件流，管理台资源卡展示最近变更。
轻表达偏好事件历史第一版已接入，聊天中关闭/克制/恢复轻表达和资料页三档设置都会写入事件流，管理台表达面板展示最近偏好变化。
偏好切换调参提示第一版已接入，管理台会在最近轻表达偏好多次切换时提示先观察用户频率反馈，避免盲目提高触发。

后续再做：

- 将表达选择小 agent 接入更细的长期反馈调参。
- 更细的情绪场景频率策略，例如从规则分类升级为基于长期反馈的调参。
- 更完整的管理员审查流程，例如审查记录批注和资源生命周期历史。

### 第四阶段及以后：多 agent、模型路由和主动联系

不要现在硬拆复杂多 agent。随着功能成熟再增加：

- 聊天主体 agent。
- 记忆抽取 / 审查 agent。
- 人格成长调整模块（当前内部名 `Sculptor`）。
- 表达/表情调度 agent。
- 安全和边界 agent。
- 主动联系系统：未来人格可以在用户离线、退出聊天或长时间未互动后主动发来消息。它不应只是一个 agent，而是多模块/多 agent 多轮排查：机会发现、时机判断、记忆依据核查、消息生成、风险边界审查、最终仲裁。
- 成本路由器：根据任务难度、重要性、上下文长度、失败率选择不同 LLM。

## 后阶段必须解决的底层问题

### 人格主体性

当前只是通过 prompt 和输出清洗禁止身份泄漏。后续要更系统地拆分：

- 系统层知道它由 AI 驱动。
- 人格层只作为对话对象自然行动。
- 聊天主体不解释系统、不暴露模型、不说“作为 AI”。
- 需要建立 Persona Runtime Contract，约束人格在对话中的自我认知、边界和表达方式。

### AI 腔

这不仅和模型有关，也和程序结构有关。后续需要：

- 更强的对话风格约束。
- 反客服腔规则。
- 少总结、少说教、少模板安慰。
- 更多基于关系状态的自然回应。
- 针对“回答不满意”的问题，不只换模型，还要完善人格生成、人格成长、关系状态和回复风格契约。
- 可能增加后处理 judge 或 style critic，但不能影响速度太多。

### 表达和贴纸

括号动作不是最终形态。最终应该是：

```text
模型自然说话
-> 表达标签 / 情绪标签
-> 表情/动作 agent 判断是否需要
-> 资源库选择贴纸、头像表情或动作
-> 前端展示
```

表达系统最终会和人物形象、头像表情、贴纸、声音和实时互动联动，而不是只把括号动作换成一行标签。

### 养成反馈与长期吸引力

这是平台能否长期留住用户的核心，不是简单 UI 装饰。

后续需要逐步设计：

- 关系状态可感知：用户能感到关系在变化，但不能变成游戏数值面板。
- 人格成长可感知：人格的说话方式、关注点、边界和亲密度会因长期互动而变化。
- 用户行为有反馈：用户的陪伴、选择、表达和长期习惯能被系统自然回应。
- 反馈要克制：不要频繁弹成就、等级或系统提示，优先让变化出现在聊天和资料细节里。
- 避免纯舒适但无目标感：陪伴要温柔，也要有新鲜感、期待感和继续打开的理由。

### 主动联系系统

这是未来增强关系生命感的重要系统，不在当前立即实现，但要作为独立平台能力规划。

核心原则：

- 用户许可是底线：用户必须能开启/关闭主动消息，设置安静时段、频率上限和敏感边界。
- 主动消息要直接以人格口吻发出内容，不使用“某某好像有话想和你说”这种系统提示。
- 主动联系必须有理由：来自记忆、状态、未完成话题、用户计划、情绪变化、长期未联系或关系阶段，而不是随机刷存在感。
- 频率要从低到高学习：初期极克制，只在强理由时发送；用户正反馈后略微增加，忽略或反感后立刻降低。
- 不能操控用户：避免制造焦虑、依赖、内疚或过度亲密压力。

建议模块/agent：

- Opportunity Scout：扫描是否存在值得主动联系的机会，例如未完成话题、计划提醒、长时间未联系、用户兴趣点。
- Context Judge：判断当前是否适合联系，例如用户许可、安静时段、近期忽略率、频率上限、风险状态。
- Memory Relevance Agent：核查主动消息是否有真实记忆依据，避免编造“我记得你说过”。
- Message Writer：生成短、自然、人格化的一句话或少量短句。
- Risk / Boundary Critic：检查过度亲密、施压、敏感话题、现实边界和安全问题。
- Policy Arbiter：最终决定发送、延后、不发或重写。
- Reaction Tracker：记录用户是否打开、回复、忽略、反感或正向评价，用于后续策略学习。

策略学习：

- 可从 contextual bandit 开始，而不是一开始上复杂深度强化学习。
- “arm” 可以是不同主动联系类型：延续话题、轻问候、情绪关心、计划提醒、兴趣话题、长时间未联系轻触达等。
- 上下文特征包括活跃时间、关系阶段、最近情绪、未读/忽略率、人格风格、用户偏好、历史响应。
- 奖励信号包括快速回复、普通回复、打开不回、长期忽略、明确反感、明确喜欢或夸赞。

阶段安排：

- 第一版：站内主动消息，进入主界面时直接显示人格发来的内容和未读提示；不做系统提示，不做系统外推送。
- 第二版：候选池和管理台审查，能看到“为什么想发、准备发什么、风险判断是什么”。
- 第三版：基于用户反馈调整主动消息类型和频率。
- 第四版：浏览器通知、PWA 推送或手机推送。
- 第五版：和声音、形象、视频等多模态表达联动。

### 形象、声音和实时陪伴

这些属于后阶段，不在当前单人格文字聊天打磨期展开，但要提前在资料字段和架构上留口。

远期方向：

- 接入擅长生图的 AI，为人格生成头像、半身像、立绘或完整形象。
- 建立人格形象档案：外貌描述、服装风格、表情风格、动作习惯、禁忌元素。
- 建立表情和贴纸资源库，让表达 agent 选择合适资源。
- 接入声音合成，形成稳定的人格声线。
- 支持语音聊天，让人格能听和说。
- 更远期支持视频电话或实时形象驱动，但必须在聊天体验、人格稳定性和成本路由成熟后再做。

### 多设备访问、部署和打包

短期开发体验：

- 本机开发继续用 `127.0.0.1:8001` 或 `127.0.0.1:8000`。
- 跨城市/跨省设备访问不能靠局域网 IP，需要公网入口。
- 临时远程测试可以用 Cloudflare Tunnel / ngrok / frp 这类隧道，把本机 `8001` 暴露成临时 HTTPS 地址。
- 当前已有脚本 `scripts/start_remote_tunnel.ps1`，会自动下载 `tools/cloudflared.exe` 并打印 `https://*.trycloudflare.com` 临时访问地址。
- 当前临时隧道测试已跑通；链接会随 cloudflared 重启变化，不要把临时链接当固定入口。
- 临时隧道适合作者自己或少数测试者体验，不适合作为成熟产品入口。
- 不要把开发服务器裸露到公网。

中期让“所有人都能打开网页”：

- 部署到 VPS / 云服务器 / PaaS，使用正式域名和 HTTPS。
- FastAPI 后面接 Nginx / Caddy / 云平台反向代理。
- SQLite 适合当前本地开发；正式多人使用时应迁移 PostgreSQL，后续向量检索可接 pgvector。
- 需要补充生产级能力：环境变量管理、日志、备份、限流、错误监控、管理员权限保护、用户数据导出/删除、隐私协议。
- 游客模式正式化前要补：更清晰的游客数据提示、到期前提醒、管理员查看游客数量和清理日志。

打包方向：

- 短期优先网页，不急着打包桌面端。
- 如果需要桌面应用，可以考虑 Tauri 或 Electron，把现有网页作为前端壳，后端仍可走远程服务器。
- 如果需要手机端，先把网页做成 PWA 或响应式移动端；成熟后再考虑 React Native / Flutter。
- 打包程序本身不能解决远程多设备同步；两省之间访问同一账号和同一人格，核心仍然是公网后端和统一数据库。
- 不建议现在把本地 SQLite + 本地后端直接打包给普通用户，因为更新、数据同步、账号体系和模型密钥管理都会变复杂。

## 开发原则

- 不破坏现有记忆系统。
- 文件编辑用 `apply_patch`。
- Windows PowerShell 环境。
- 搜索用 `rg`。
- 改完跑 compile/check。
- UI 不做营销 landing page，第一屏就是聊天体验。
- 不为了短期 demo 牺牲长期地基。
- 用户不会设计前端，所以开发者要主动判断交互和布局。
- 每次做完重要变化，更新本文件或 README，保证换聊天也能接上。
